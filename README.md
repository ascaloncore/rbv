# RBV (Reading BookValue)
〜 知識の「不良在庫」をゼロにし、自己投資のROIを最大化する学習資産管理アプリ 〜

## サービス概要
RBV (Reading BookValue) は、購入したものの未消化となっている学習教材(いわゆる、積ん読や未読本)を、企業の「棚卸資産（在庫）」に見立てて管理するアプリです。
一般的な学習記録（時間管理）ではなく、「購入金額（コスト）」と「在庫状態(適正在庫・余剰在庫・滞留在庫・不良在庫などのステージ)」を可視化し、埋没費用（損失）をユーザーに対して突きつけることで学習動機を生み出します。
「知識の不良在庫」をゼロにし、自己投資のROI（投資対効果）を最大化することを目的としています。

## このサービスへの思い・作りたい理由
### 学習資産（財政状態）の「見える化」を通じたROIの最大化
私は20年間、経理財務プロフェッショナルとして企業のディスクロージャー（決算開示）に関わる中、特にグループの財政状態の重要性を肌身で感じております。財政状態は、「現金への回収可能額」を表現したものでもあり、企業は調達した負債（借入金）と資本（株式）により、投資リターンを最大化させ、財政状態及び経営成績に一連の企業活動の結果を表現した上、厳しい説明責任が求められます。
特に資産側である「棚卸資産」における「現金への回収可能価額」は、生産性指標（KPI）である「在庫回転率」や「不良在庫率」などの財務分析を通じて、内部管理会計及び外部監査人などの利害関係者により厳しく評価されますが、これを私個人の学習資産に置き換えた場合、私の学習資産はまさに「経営破綻」状態でした。
「いつかやる」と買い込んだ技術書、セールの勢いで買った動画講座……これらは本棚やHDDの肥やしとなり、財務分析的視点で見れば「巨額の不良在庫」を抱え、キャッシュフローが悪化している状態でした。

**「自分という会社の経営なら、これは即刻倒産だ」**

この強烈な危機感が、このアプリの出発点です。
既存の学習アプリは「頑張った記録」をつけるものばかりで、「無駄にしている損失」を直視させるものはありませんでした。
多忙な現代人が「積読（つんどく）」の罪悪感から脱出し、投じたコストを確実に「血肉（資産）」に変えるためには、精神論ではなく、企業会計的・経営的なアプローチによる管理が必要だと確信し、開発を決意しました。

## ユーザー層について
### ターゲット
20代〜40代の読書習慣のある学生・エンジニア・ビジネスパーソン

### この層を対象にした理由
この層は自己成長意欲が高く、書籍代や研修費に月1万円以上を投資する経済力があります。しかし、仕事が多忙であるため「購入したことで満足する」という罠に陥りがちです。
彼らに最も響くのは、「頑張りましょう」という感情的な励ましではなく、**「現在、あなたは24,000円をドブに捨てています」という合理的かつ経済的なファクト（事実）**であると考えたためです。

## サービスの利用イメージ
本アプリは、ユーザーの学習状況を**B/S（貸借対照表）**のように管理し、「損失回避」「赤字脱却」をゴールに行動変容を促します。

1. **衝撃 (Reality Check)**
ログイン直後のダッシュボードに、未完了教材の総額が**「投下資産」のうち、「埋没費用（不良在庫）」**として大きく表示されます。ユーザーは自身の浪費状況を突きつけられ、焦燥感を覚えます。
2. **仕入 (Asset Entry)**
購入した教材を登録します。あえてAPIによる自動取得ではなく手入力を基本とすることで、出費の痛みを自覚させます。この時点でのステータスは「適正在庫」です。
3. **資産化 (Transformation)**
学習を開始すると「仕掛品（WIP:Work in Progress）」へ、完了すると「知識資産（自己投資の成功）」へステージングアップされ、投資完了となります。学習の完了をもって初めて、その購入金額が「知識資産」として計上され、ダッシュボードの「埋没費用（不良在庫）」が減少します。
4. **決算 (Closing)**
ユーザーは、四半期または年度締めイベントにおいて、「棚卸資産」の実地棚卸を実施します。この手続き中で「不良在庫」や政策的に「回収可能額」が乏しい「在庫」や「仕掛品」を「棚卸評価減」または「減損損失（事業撤退）」をすることで「埋没費用」を減少させ、自己のROIの最大化に向けた新たな決意と無駄にした投下資本を認識します。
5. **価値 (Value)**
ユーザーは「不良在庫による損失（赤字）を回避したい」という動機で学習を進め、結果として知識という資産を手に入れます。

## ユーザーの獲得について
### 想定ユーザーへのアプローチ
* **エンジニアコミュニティ（RUNTEQ等）への展開**
「在庫一掃キャンペーン」と銘打ち、年末年始や年度末などの節目に、自身の積読状況を棚卸しするイベントとして利用を促します。
* **SNS（X/Twitter）でのバイラル**
「私の積読損失額は〇〇円でした #RBV」という、自虐的かつインパクトのあるシェア機能を実装し、エンジニア界隈での話題化を狙います。「学習時間」のシェア（ポジティブ）に対するアンチテーゼとして、「損失額」のシェア（ネガティブ・ユニーク）を定着させます。

## サービスの差別化ポイント・推しポイント
### 競合アプリ（Studyplus等）との差別化

| 比較軸 | 既存アプリ（Studyplus等） | 本アプリ (RBV) |
| --- | --- | --- |
| **管理対象** | 学習時間・継続日数 (Process) | 購入金額・在庫状態 |
| **視点** | フロー（どれだけ時間を流したか） | ストック（どれだけ資産が積み上がったか） |
| **動機付け** | ポジティブ（積み上げ・いいね） | ネガティブ（損失回避・赤字解消） |

### 推しポイント：徹底した「経理メタファー」の世界観
* 在庫のステータス管理を「未着手」「仕掛品」「知識資産」「余剰在庫」「滞留在庫」「不良在庫」など会計的用語で定義。
* 事務的になりがちな学習タスク消化を、「ROI最大化」というミッションに変換し、ユーザーを「経営者視点」に引き上げます。
* 「引き算の美学」として、SNS機能やAPI連携をあえて排除し、孤独に数字と向き合う硬派なツールとしての立ち位置を確立します。

## 機能候補
### MVPリリース時に作っていたいもの
**「赤字の可視化による動機付け」を検証するための最小機能**
* **ユーザー認証機能** (Sorcery)
* **教材資産管理機能 (CRUD)**
  * タイトル、金額、購入日、カテゴリ（書籍/動画等）の登録・編集・削除。
* **棚卸しダッシュボード機能【核となる機能】**
  * 未完了教材（各ステージの在庫）の合計金額を自動計算し、トップページに強調表示。
* **ステータス管理機能**
  * 未着手 → 学習中 → 完了（資産化） → 挫折（棚卸資産評価減または減損損失）のステータス遷移。
* **学習メモ機能**
  * 各教材に対する簡易的な学習ログの記録。

### 本リリースまでに作っていたいもの
* **評価減アラート:** 購入から一定期間（例: 3ヶ月）放置された教材を、強制的に「余剰在庫」として警告通知。
* **決算報告（SNSシェア）:** 「今月の私の知識資産残高は〇〇円、不良在庫はゼロです。また、棚卸資産評価減として〇〇円計上しました #RBV」というOGP画像を生成しシェア。
* **バーコード/URL連携:** ISBNスキャンやUdemyのURL貼り付けによる、書籍名・価格の自動取得。

## 使用する技術スタック
### 実装機能のイメージ（CRUD以外）
* **ダッシュボード集計ロジック:**
ユーザーに紐づく教材データを取得し、ステータス（在庫）に基づいてSQLまたはRuby側で合計金額を算出。ActiveRecordのScopeを活用して「赤字」と「資産」を分類します。
* **ステータス遷移のUI:**
Hotwire (Turbo Frames/Streams) を使用し、一覧画面や詳細画面でステータスを変更した際、ページ遷移なしでダッシュボードの金額が更新されるSPA風の挙動を実装します。
